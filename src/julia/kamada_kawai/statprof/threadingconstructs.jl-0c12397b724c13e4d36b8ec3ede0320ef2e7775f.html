259 """
260 Threads.@threads [schedule] for ... end
261
262 A macro to execute a `for` loop in parallel. The iteration space is distributed to
263 coarse-grained tasks. This policy can be specified by the `schedule` argument. The
264 execution of the loop waits for the evaluation of all iterations.
265
266 See also: [`@spawn`](@ref Threads.@spawn) and
267 `pmap` in [`Distributed`](@ref man-distributed).
268
269 # Extended help
270
271 ## Semantics
272
273 Unless stronger guarantees are specified by the scheduling option, the loop executed by
274 `@threads` macro have the following semantics.
275
276 The `@threads` macro executes the loop body in an unspecified order and potentially
277 concurrently. It does not specify the exact assignments of the tasks and the worker threads.
278 The assignments can be different for each execution. The loop body code (including any code
279 transitively called from it) must not make any assumptions about the distribution of
280 iterations to tasks or the worker thread in which they are executed. The loop body for each
281 iteration must be able to make forward progress independent of other iterations and be free
282 from data races. As such, invalid synchronizations across iterations may deadlock while
283 unsynchronized memory accesses may result in undefined behavior.
284
285 For example, the above conditions imply that:
286
287 - A lock taken in an iteration *must* be released within the same iteration.
288 - Communicating between iterations using blocking primitives like `Channel`s is incorrect.
289 - Write only to locations not shared across iterations (unless a lock or atomic operation is
290 used).
291 - Unless the `:static` schedule is used, the value of [`threadid()`](@ref Threads.threadid)
292 may change even within a single iteration. See [`Task Migration`](@ref man-task-migration).
293
294 ## Schedulers
295
296 Without the scheduler argument, the exact scheduling is unspecified and varies across Julia
297 releases. Currently, `:dynamic` is used when the scheduler is not specified.
298
299 !!! compat "Julia 1.5"
300 The `schedule` argument is available as of Julia 1.5.
301
302 ### `:dynamic` (default)
303
304 `:dynamic` scheduler executes iterations dynamically to available worker threads. Current
305 implementation assumes that the workload for each iteration is uniform. However, this
306 assumption may be removed in the future.
307
308 This scheduling option is merely a hint to the underlying execution mechanism. However, a
309 few properties can be expected. The number of `Task`s used by `:dynamic` scheduler is
310 bounded by a small constant multiple of the number of available worker threads
311 ([`Threads.threadpoolsize()`](@ref)). Each task processes contiguous regions of the
312 iteration space. Thus, `@threads :dynamic for x in xs; f(x); end` is typically more
313 efficient than `@sync for x in xs; @spawn f(x); end` if `length(xs)` is significantly
314 larger than the number of the worker threads and the run-time of `f(x)` is relatively
315 smaller than the cost of spawning and synchronizing a task (typically less than 10
316 microseconds).
317
318 !!! compat "Julia 1.8"
319 The `:dynamic` option for the `schedule` argument is available and the default as of Julia 1.8.
320
321 ### `:greedy`
322
323 `:greedy` scheduler spawns up to [`Threads.threadpoolsize()`](@ref) tasks, each greedily working on
324 the given iterated values as they are produced. As soon as one task finishes its work, it takes
325 the next value from the iterator. Work done by any individual task is not necessarily on
326 contiguous values from the iterator. The given iterator may produce values forever, only the
327 iterator interface is required (no indexing).
328
329 This scheduling option is generally a good choice if the workload of individual iterations
330 is not uniform/has a large spread.
331
332 !!! compat "Julia 1.11"
333 The `:greedy` option for the `schedule` argument is available as of Julia 1.11.
334
335 ### `:static`
336
337 `:static` scheduler creates one task per thread and divides the iterations equally among
338 them, assigning each task specifically to each thread. In particular, the value of
339 [`threadid()`](@ref Threads.threadid) is guaranteed to be constant within one iteration.
340 Specifying `:static` is an error if used from inside another `@threads` loop or from a
341 thread other than 1.
342
343 !!! note
344 `:static` scheduling exists for supporting transition of code written before Julia 1.3.
345 In newly written library functions, `:static` scheduling is discouraged because the
346 functions using this option cannot be called from arbitrary worker threads.
347
348 ## Examples
349
350 To illustrate of the different scheduling strategies, consider the following function
351 `busywait` containing a non-yielding timed loop that runs for a given number of seconds.
352
353 ```julia-repl
354 julia> function busywait(seconds)
355 tstart = time_ns()
356 while (time_ns() - tstart) / 1e9 < seconds
357 end
358 end
359
360 julia> @time begin
361 Threads.@spawn busywait(5)
362 Threads.@threads :static for i in 1:Threads.threadpoolsize()
363 busywait(1)
364 end
365 end
366 6.003001 seconds (16.33 k allocations: 899.255 KiB, 0.25% compilation time)
367
368 julia> @time begin
369 Threads.@spawn busywait(5)
370 Threads.@threads :dynamic for i in 1:Threads.threadpoolsize()
371 busywait(1)
372 end
373 end
374 2.012056 seconds (16.05 k allocations: 883.919 KiB, 0.66% compilation time)
375 ```
376
377 The `:dynamic` example takes 2 seconds since one of the non-occupied threads is able
378 to run two of the 1-second iterations to complete the for loop.

417 """
418 Threads.@spawn [:default|:interactive] expr
419
420 Create a [`Task`](@ref) and [`schedule`](@ref) it to run on any available
421 thread in the specified threadpool (`:default` if unspecified). The task is
422 allocated to a thread once one becomes available. To wait for the task to
423 finish, call [`wait`](@ref) on the result of this macro, or call
424 [`fetch`](@ref) to wait and then obtain its return value.
425
426 Values can be interpolated into `@spawn` via `\$`, which copies the value
427 directly into the constructed underlying closure. This allows you to insert
428 the _value_ of a variable, isolating the asynchronous code from changes to
429 the variable's value in the current task.
430
431 !!! note
432 The thread that the task runs on may change if the task yields, therefore `threadid()` should not
433 be treated as constant for a task. See [`Task Migration`](@ref man-task-migration), and the broader
434 [multi-threading](@ref man-multithreading) manual for further important caveats.
435 See also the chapter on [threadpools](@ref man-threadpools).
436
437 !!! compat "Julia 1.3"
438 This macro is available as of Julia 1.3.
439
440 !!! compat "Julia 1.4"
441 Interpolating values via `\$` is available as of Julia 1.4.
442
443 !!! compat "Julia 1.9"
444 A threadpool may be specified as of Julia 1.9.
